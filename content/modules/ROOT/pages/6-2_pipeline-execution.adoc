= Module 5.2: Execute retraining pipeline

The retraining pipeline automates the complete model lifecycle: data collection → preprocessing → training → validation → conditional deployment. Execute the pipeline manually to understand the automated workflow.

== Learning objectives

By the end of this module, you will be able to:

* Import a pre-built pipeline definition from YAML
* Execute a pipeline run manually with configured parameters
* Monitor pipeline execution stages in real-time
* Verify conditional deployment logic (only deploy improved models)

[#understanding-retraining-pipeline]
== Understanding the retraining pipeline

The pipeline automates:

1. **Data Collection** - Retrieve fresh battery telemetry from InfluxDB on robots
2. **Data Preparation** - Process and normalize data for training
3. **Model Training** - Train Stress Detection and Time-to-Failure models in parallel
4. **Model Validation** - Compare new models against existing models using test data
5. **Conditional Deployment** - If new models outperform existing ones, upload to robot storage

This workflow ensures models stay accurate as robot battery behavior evolves.

[#import-pipeline]
== Exercise 1: Import the retraining pipeline

Instead of creating the pipeline from scratch, import a pre-built definition.

**Download Pipeline Definition:**

1. Open the **JupyterLab** workbench (model-training)
2. Navigate to `ai-lifecycle-edge-gitops/notebooks/pipelines/`
3. Locate the `model-retraining.yaml` file
4. **Right-click** on it, select **Download**

IMAGE_HERE: JupyterLab right-click context menu showing Download option for model-retraining.yaml

5. Save the file to your laptop

**Import Pipeline:**

1. Switch to the **Red Hat OpenShift AI (RHOAI) Dashboard**
2. Navigate to the **Pipelines** tab in `ai-edge-project`
3. Click **Import pipeline**
4. Fill in the form:
   * **Project**: `ai-edge-project` (pre-selected)
   * **Pipeline name**: `Model retraining`
   * **Upload a file**: Ensure this option is checked
   * Click **Upload**, select the `model-retraining.yaml` file from your laptop

5. Click **Import pipeline**

The pipeline graph displays, showing nodes and their connections.

IMAGE_HERE: Pipeline graph view showing data retrieval, preparation, parallel training, validation, and deployment nodes

== Exercise 3: Monitor pipeline execution

1. **Data Retrieval** - Fetches battery telemetry from InfluxDB on robots
2. **Data Processing** - Prepares and normalizes data
3. **Parallel Training** - Both models train simultaneously
4. **Model Validation** - New models evaluated against existing models

[#verify-pipeline-success]
== Exercise 4: Verify pipeline success

After execution completes, verify the pipeline ran successfully.

=== Verify

**Check Pipeline Status:**

✓ All nodes show **green checkmarks** (successful completion)
✓ No nodes show **red X** (failed status)
✓ Pipeline run shows "Succeeded" status in the Runs tab
✓ Run duration is reasonable (typically 5-15 minutes)

**View Execution Logs:**

✓ Can click any node in graph to view execution logs
✓ Logs show detailed execution steps for that stage
✓ No error messages in critical stages (training, validation, deployment)
✓ Each stage reports successful completion

**Check Model Deployment (if models improved):**

If new models outperformed existing ones, verify they were uploaded to MinIO:

1. Open MinIO dashboard: `https://minio-microshift-001.{openshift_cluster_ingress_domain}`
2. Login: `minio` / `minio123`
3. Navigate to `inference` bucket
4. Check model files in `stress-detection/1/` and `time-to-failure/1/`

✓ Model files (`model.xml`, `model.bin`) exist
✓ File timestamps show recent updates (within last few minutes)
✓ Both model directories updated if retraining improved both models
✓ Deployment node in pipeline shows successful upload

**If models did NOT improve:**
- Deployment node may show "No deployment needed"
- This is expected and correct behavior (keep existing better models)
