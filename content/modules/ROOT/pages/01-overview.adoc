[#the-challenge]
== The challenge: Factory robot battery monitoring

Batteries power modern industrial automation. In automotive manufacturing and logistics facilities, battery-powered autonomous mobile robots (AMRs) have become essential for efficient operations. Our facility operates a **fleet of 18 battery-powered robots** that transport pallets of automotive components between receiving docks, warehouse storage, production lines, and shipping areas **24/7**.

Batteries are the heart of these machines. An unexpected battery failure can lead to:

* Immediate production delays when robots stop mid-route
* Disrupted just-in-time delivery schedules
* Safety risks if robots fail in critical areas
* Costly unplanned downtime and maintenance

**This is where Artificial Intelligence enables predictive maintenance.**

[#the-solution]
== Our AI-powered solution

As systems engineers for an automotive parts distribution center, we've developed an end-to-end AI solution that covers two scenarios:

=== Battery monitoring system

Each factory robot runs an application that monitors battery health during operational routes throughout the facility. This application:

* Displays real-time telemetry data (voltage, temperature, charge cycles, operational distance) via a dashboard accessible from the plant control center
* Uses AI models to detect early signs of battery stress
* Predicts potential battery failure before it happens
* Triggers alerts to maintenance teams

To support this containerized solution on each robot, we use **Red Hat Device Edge (MicroShift)**, a lightweight Kubernetes platform optimized for resource-constrained environments.

=== Automated model re-training at the edge

AI models must stay accurate as battery behavior evolves over time. However, robots generate massive amounts of telemetry data (40-60 trips per shift, 24/7 operations), making it impractical to continuously upload all data over the factory network.

Our solution:

* Collects new data every 10 minutes from the robot fleet
* Retrains AI models on an edge server with greater compute capacity (Single Node OpenShift) at the plant
* Uses **Red Hat OpenShift AI (RHOAI)** for the entire AI/ML lifecycle
* Runs a fully automated pipeline: data collection → model training → validation → deployment
* Automatically deploys improved models back to robots if they outperform existing ones

This ensures smarter battery monitoring without manual intervention.

[#edge-computing-essentials]
== Edge computing essentials

Edge computing brings computation and data storage closer to where data is generated, rather than relying on centralized cloud data centers. For our factory robots, this means:

**Why Edge Matters:**

* **Low latency** - Real-time battery predictions happen instantly on the robot
* **Reduced bandwidth** - Only aggregated data is sent to the training server, not continuous streams
* **Reliability** - Robots continue operating even if network connectivity is lost
* **Data privacy** - Sensitive operational data stays within the factory
* **Cost efficiency** - Less data transfer to cloud systems

**Red Hat Edge Platforms:**

* **MicroShift** - Lightweight Kubernetes for resource-constrained devices (robots)
** Minimal footprint (~2 CPU cores, 2GB RAM)
** Runs on RHEL 9.6
** Includes model serving and GitOps capabilities
** Perfect for distributed robot fleet deployments

* **Single Node OpenShift (SNO)** - Full OpenShift on a single machine (training server)
** Edge server with greater compute capacity located at the plant
** Runs Red Hat OpenShift AI for model training and pipeline automation
** Optimized for edge locations without dedicated IT teams
** Provides enterprise Kubernetes features in a single-node deployment

[#architecture]
== Solution architecture

Our infrastructure consists of two key components working together:

=== Factory Robot (MicroShift on RHEL 9.6)

Each robot runs:

* **MicroShift** - Lightweight Kubernetes platform
* **MinIO Storage** - Stores AI models in the `inference` bucket
* **Model Serving** - KServe/ModelMesh for loading and serving models via API
* **Battery Monitoring System (BMS)** - Dashboard showing real-time telemetry and AI predictions
* **InfluxDB** - Time-series database storing historical sensor data
* **MQTT Broker** - Receives telemetry from battery simulators
* **Battery Simulator** - Quarkus app generating realistic battery consumption data

=== Re-training server (Single Node OpenShift)

Located at the plant:

* **Red Hat OpenShift AI** - Complete AI/ML platform
** Data science projects and workbenches (JupyterLab)
** Model training with TensorFlow
** Automated pipelines for retraining workflow
** Model serving for validation
** TrustyAI for model monitoring

* **MinIO Storage** - Stores pipeline artifacts in the `pipelines` bucket

=== Workflow diagram

The diagram below illustrates the complete solution:

IMAGE_HERE: Architecture diagram showing robots, MicroShift, SNO, and data flow

**Workflow Steps:**

1. **Base Models** - Two AI models stored in MinIO: *Stress Detection* (identifies battery stress) and *Time to Failure* (predicts remaining battery life)

2. **Model Serving** - Models loaded into InferenceServer, available via API endpoints for predictions

3. **Data Generation** - Battery simulator generates telemetry data, sends to MQTT broker

4. **Data Processing** - Camel Quarkus components:
** `mqtt2ws` - Exposes data as WebSocket for BMS dashboard
** `data-ingester` - Stores data in InfluxDB

5. **Real-Time Predictions** - BMS retrieves data from InfluxDB, queries inference endpoints, displays predictions

6. **Alerting** - If models detect stress or predict failure, alerts are triggered

7. **Automated Retraining** - Every 10 minutes, pipeline on SNO:
** Collects last hour of data from InfluxDB
** Retrains both models
** Validates new models vs. existing models
** Compares performance

8. **Conditional Deployment** - If new models outperform existing ones, they're automatically uploaded to robot's MinIO storage, and the cycle continues
