= Module 4.1: Review Stress Detection Server

Model servers provide real-time inference capabilities by loading trained models and exposing REST API endpoints. The Stress Detection server validates newly trained models before deployment to robots.

[#understanding-model-serving]
== Understanding model serving in RHOAI

**Why Model Serving on SNO?**

Although models run on robots for production inference, the SNO environment also serves models for:

* **Validation** - Test newly trained models before deploying to robots
* **Comparison** - Benchmark new models against existing ones
* **Development** - Quick testing during model development

**KServe Deployment Modes:**

* **Serverless** - Auto-scaling with Knative (not used here)
* **RawDeployment** - Manual configuration, predictable resources (used for edge)

RawDeployment provides full control without auto-scaling overhead, ideal for resource-constrained SNO.

== Task 4: Review ServingRuntime and InferenceService

Switch to the **OpenShift Web Console** to examine underlying resources.

**ServingRuntime:**

1. Navigate to **Search** in the left navigation
2. In the **Resources** dropdown, search for `ServingRuntime`
3. Ensure you're in the `ai-edge-project` namespace
4. Click on the ServingRuntime for OpenVINO Model Server

The ServingRuntime YAML defines:

* Container image for OpenVINO Model Server
* Supported model formats (openvino_ir, ONNX, TensorFlow, PyTorch)
* Runtime parameters and environment variables

**InferenceService:**

1. In the **Resources** dropdown, search for `InferenceService`
2. Find the `stress-detection` InferenceService
3. Click to view the YAML

The InferenceService YAML specifies:

* Reference to ServingRuntime (OpenVINO)
* Model storage location (`vehicle-models` connection, `stress-detection` path)
* Resource requests and limits
* Scaling parameters (replicas: 1 for RawDeployment)
