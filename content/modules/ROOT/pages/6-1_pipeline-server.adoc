= Module 5.1: Review pipeline server

Automation is critical for edge deployments where IT teams aren't readily available. Red Hat OpenShift AI pipelines automate the complete ML workflow: data collection → training → validation → deployment.

== Learning objectives

By the end of this module, you will be able to:

* Understand why pipeline automation is critical for edge deployments
* Verify the pipeline server is running and accessible
* Review pipeline storage configuration and DataSciencePipelineApplication
* Identify pipeline server components (API server, scheduler, persistence agent)

[#understanding-pipeline-automation]
== Understanding pipeline automation

**Why Automation Matters at the Edge:**

* **No manual intervention** - Pipelines run autonomously every 10 minutes
* **Self-sustaining** - Models continuously improve without human oversight
* **Reliable** - Automated workflows maintain system stability
* **Efficient** - Reduces downtime and optimizes performance

**Kubeflow Pipelines 2.0:**

Red Hat OpenShift AI (RHOAI) implements Kubeflow Pipelines for ML workflow automation. Pipelines define:

* **Steps** - Data extraction, preprocessing, training, validation, deployment
* **Inputs** - Source data, configuration parameters
* **Output artifacts** - Trained models, metrics, logs (stored in S3)

[#review-pipeline-storage]
== Exercise 2: Review pipeline storage connection

The pipeline server uses dedicated storage for execution artifacts.

1. In the **Pipelines** tab, note the pipeline server configuration
2. Navigate to the **Connections** tab
3. Locate the `pipeline-storage` DataConnection (alongside `vehicle-models`)

Review connection details:

* **Connection type**: S3 compatible object storage
* **Endpoint**: `http://minio-service.minio.svc.cluster.local:9000` (MinIO on SNO)
* **Region**: `us-east-1`
* **Bucket**: `pipelines` (dedicated bucket for pipeline artifacts)

**Storage Separation:**

* **vehicle-models** → Robot's MinIO (models and training data)
* **pipeline-storage** → SNO's MinIO (execution logs, metrics, intermediate data)

This separation keeps pipeline artifacts isolated from production model storage.

=== Verify

Confirm pipeline storage connection exists:

✓ **Connections** tab shows both `vehicle-models` and `pipeline-storage`
✓ **pipeline-storage** connection type: S3 compatible object storage
✓ **Endpoint**: Points to SNO's MinIO service (`minio-service.minio.svc.cluster.local:9000`)
✓ **Bucket**: `pipelines` (dedicated for execution artifacts)
✓ Connection is separate from `vehicle-models` (robot storage)

[#understanding-pipeline-storage]
== Understanding what pipelines store

The `pipelines` bucket stores:

* **Pipeline definitions** - Compiled workflow YAML files
* **Execution artifacts** - Logs, metrics, intermediate outputs
* **Model metrics** - Training accuracy, loss curves, validation scores
* **Comparison data** - Performance comparisons between model versions

This data enables:

* **Reproducibility** - Re-run pipelines with identical parameters
* **Debugging** - Analyze failed pipeline runs
* **Metrics tracking** - Monitor model performance over time
* **Audit trail** - Track all training and deployment activities
