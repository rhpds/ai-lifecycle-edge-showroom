= Module 2: RHOAI Configuration

Red Hat OpenShift AI (RHOAI) running on Single Node OpenShift provides the complete AI/ML platform for model training, serving, pipelines, and monitoring. This module reviews the pre-configured RHOAI environment optimized for edge deployments.

=== Review DataScienceCluster configuration

The DataScienceCluster custom resource defines which RHOAI components are enabled and how they're configured.

. In **Installed Operators**, click **Red Hat OpenShift AI**
. Navigate to the **DataScienceCluster** tab
. Click the existing DataScienceCluster instance
. Click the **YAML** tab to view the configuration

**Key configuration points:**

**KServe Configuration:**

* `defaultDeploymentMode: RawDeployment` - Full control over Kubernetes resources without Knative overhead
* `managementState: Managed` - KServe is enabled and managed by the operator

**Why RawDeployment for Edge?**

* Full control over Kubernetes resources without Knative overhead
* Ideal for resource-constrained SNO environments
* Predictable resource usage (no auto-scaling)
* Simpler configuration for edge deployments

**KNative-Serving Configuration:**

* `managementState: Removed` - KNative stack is disabled

**Why Remove KNative?**

* RawDeployment doesn't use KNative
* Reduces resource consumption on SNO
* Eliminates unnecessary components

**TrustyAI Configuration:**

* `managementState: Managed` - TrustyAI is enabled for model monitoring

**Why Enable TrustyAI?**

* Monitors model bias and fairness
* Detects data drift over time
* Essential for automated retraining (models continuously update without human oversight)

[#review-dataconnection]
== Exercise 2.3: Review DataConnection

DataConnections link RHOAI to external object storage. The `vehicle-models` connection provides access to the robot's MinIO storage where AI models and training data are stored.

**DataConnections enable:**

* **Workbenches** - Access training data from robot storage
* **Model Servers** - Load models for validation
* **Pipelines** - Read input data and write trained models

The connection bridges the SNO training environment and robot's edge storage.

=== Review the vehicle-models DataConnection

. Ensure you're in the **RHOAI Dashboard** inside the `ai-edge-project` project
. Navigate to the **Connections** tab
. Locate the existing connection: `vehicle-models`

image::3-3_connection.png[Connections tab showing vehicle-models DataConnection,link=self,window=blank,width=700]

=== Understand connection parameters

The `vehicle-models` connection is configured with:

* **Connection type**: S3 compatible object storage - v1 (standard S3 protocol compatible with MinIO)
* **Connection name**: `vehicle-models`
* **Endpoint**: `http://minio-microshift-vm.microshift-001.svc.cluster.local:30000`
* **Region**: `us-east-1` (default S3 region, required but unused by MinIO)
* **Bucket**: `inference` (contains AI models and training data)

**Endpoint Explanation:**

The URL `minio-microshift-vm.microshift-001.svc.cluster.local:30000` uses Kubernetes service DNS to connect across clusters. The `microshift-001.svc.cluster.local` domain allows SNO to communicate with the robot's MicroShift cluster over the factory network.

=== Verify the underlying secret

DataConnections are stored as Kubernetes Secrets. Review the secret:

. Switch to the **OpenShift Web Console**
. Navigate to **Workloads** → **Secrets**
. Ensure you're in the `ai-edge-project` namespace (use the project dropdown)
. Search for or scroll to: `aws-connection-vehicle-models`
. Click on the secret to view details

The secret contains base64-encoded S3 credentials:

* `AWS_ACCESS_KEY_ID` - MinIO username
* `AWS_SECRET_ACCESS_KEY` - MinIO password
* `AWS_S3_ENDPOINT` - MinIO API endpoint
* `AWS_DEFAULT_REGION` - S3 region
* `AWS_S3_BUCKET` - Target bucket name

NOTE: All values are base64-encoded for security. RHOAI components automatically decode them when using the connection.

=== Verify

Confirm the DataConnection is properly configured:

✓ `vehicle-models` connection appears in Connections tab

✓ Connection type shows "S3 compatible object storage - v1"

✓ **Endpoint**: Points to MicroShift MinIO service

✓ **Bucket**: `inference`

✓ Secret `aws-connection-vehicle-models` exists in `ai-edge-project` namespace

✓ Secret contains all 5 required fields (base64-encoded)

[#troubleshooting]
== Troubleshooting

**Issue**: RHOAI operator shows "Failed" or "UpgradePending" status

**Solution**:

. Check operator events: Navigate to Operators → Installed Operators → Red Hat OpenShift AI → Events tab
. Review operator logs: Click on operator pod in `redhat-ods-operator` namespace → Logs tab
. Verify subscription: Operators → Installed Operators → Red Hat OpenShift AI → Subscription tab
. Common fix: Ensure cluster has sufficient resources, wait for automatic reconciliation

**Issue**: DataScienceCluster stuck in "NotReady" phase

**Solution**:

. Check DataScienceCluster status: Navigate to DataScienceCluster tab → Click instance → YAML tab
. Review conditions section at bottom of YAML for error messages
. Verify component deployments: Check pods in `redhat-ods-applications` namespace
. Common fix: Ensure KNative is removed (`serving.managementState: Removed`) for edge deployments

**Issue**: RHOAI Dashboard not accessible or shows 404 error

**Solution**:

. Verify route exists: In OpenShift Console → Networking → Routes → `redhat-ods-applications` namespace
. Check for route named `rhods-dashboard`
. Verify route target service is running
. Common fix: Access via application launcher (9 squares icon) instead of direct URL

**Issue**: Cannot access ai-edge-project in RHOAI Dashboard

**Solution**:

. Verify project exists: In OpenShift Console → Projects → search for `ai-edge-project`
. Check permissions: In RHOAI Dashboard → ai-edge-project → Permissions tab
. Verify you're logged in as cluster admin user
. Common fix: Ensure you authenticated with correct credentials (`{openshift_cluster_admin_username}`)

**Issue**: DataConnection shows error or "Connection Failed" status

**Solution**:

. Verify secret exists: OpenShift Console → Workloads → Secrets → `ai-edge-project` namespace
. Check secret name: `aws-connection-vehicle-models`
. Verify MinIO endpoint is reachable from SNO
. Test connectivity: Create test pod in `ai-edge-project`, exec into it, curl the MinIO endpoint
. Common fix: Verify network connectivity between SNO and MicroShift clusters
