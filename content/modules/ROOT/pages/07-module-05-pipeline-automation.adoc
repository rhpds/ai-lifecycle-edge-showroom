== Exercise 5.1: Review pipeline infrastructure

Kubeflow Pipelines provides workflow automation for machine learning. The pipeline server orchestrates the complete model lifecycle from data collection through deployment.

=== Understanding pipeline architecture

**Why Pipelines Matter:**

Machine learning workflows involve multiple steps that must execute in sequence:

1. **Data Collection** - Retrieve fresh telemetry from robots
2. **Data Preparation** - Process and normalize training datasets
3. **Model Training** - Train models using updated data
4. **Model Validation** - Compare new models against existing ones
5. **Conditional Deployment** - Deploy only if new models outperform

Pipelines automate this workflow, ensuring:

* **Consistency** - Same process every time
* **Auditability** - Complete execution history
* **Scalability** - Handles increasing data volumes
* **Reliability** - Automatic retries on failures

**Pipeline Components:**

* **Pipeline Server** - Orchestrates workflow execution
* **Artifact Storage** - Stores pipeline outputs and metadata
* **Database** - Tracks runs, experiments, and metrics
* **UI** - RHOAI Dashboard integration for monitoring

=== Review pipeline server deployment

. In the **Red Hat OpenShift AI (RHOAI) Dashboard**, navigate to the `ai-edge-project` project
. Click the **Pipelines** tab
. Verify the pipeline server is configured and accessible

The pipeline server was automatically deployed when the project was created with pipeline support enabled.

=== Examine pipeline custom resource

Switch to the **OpenShift Web Console** to examine the underlying configuration.

. Navigate to **Search** in the left navigation
. In the **Resources** dropdown, search for `DataSciencePipelineApplication`

The DataSciencePipelineApplication custom resource defines:

* **Pipeline server** - Workflow orchestration engine
* **Database** - MariaDB or MySQL for metadata storage
* **Object storage** - S3-compatible storage for artifacts
* **API server** - REST API for pipeline management

The pipeline automates:

1. **Data Collection** - Retrieve fresh battery telemetry from InfluxDB on robots

== Exercise 5.3: Schedule pipeline automation

Scheduled pipelines enable fully automated model retraining. Configure the pipeline to run every 10 minutes, ensuring models continuously improve without manual intervention.

=== Why scheduled automation matters

Machine learning models drift over time as operational conditions change:

* **Battery aging** - Performance degrades as batteries wear
* **Temperature variations** - Seasonal changes affect battery behavior
* **Workload patterns** - Different operational routes stress batteries differently

**Scheduled Retraining Benefits:**

* **Eliminates manual intervention** - No human involvement required
* **Reduces model drift** - Models stay accurate with fresh data
* **Ensures reliability** - Predictive accuracy maintained over time
* **Enables true edge AI** - Autonomous systems that self-improve

For our 18-robot fleet operating 24/7, automation is essential.

=== Understanding the automated workflow

Every 10 minutes, the scheduled pipeline:

. **Retrieves** - Fresh battery telemetry from factory robots (last hour of data)
. **Processes** - Normalizes and prepares training datasets
. **Trains** - New Stress Detection and Time-to-Failure models
. **Validates** - Compares new models against existing models
. **Deploys** - Uploads improved models to robot storage (if they outperform)
. **Repeats** - Cycle continues every 10 minutes

This creates a **self-improving system** where models get better over time without human intervention.

[#troubleshooting]
== Troubleshooting

**Issue**: Pipeline import fails with "Invalid YAML" error

**Solution**:

. Verify downloaded file is complete (not truncated)
. Check file extension is `.yaml` not `.yaml.txt`
. Open file in text editor to verify YAML structure
. Common fix: Re-download file from JupyterLab

**Issue**: Pipeline run fails at data retrieval stage

**Solution**:

. Verify InfluxDB is accessible from pipeline pods
. Check InfluxDB credentials in pipeline parameters
. Confirm battery simulation is generating data on robot
. Verify `influxdb_url` parameter matches actual endpoint
. Common fix: Update `influxdb_url` to correct domain

**Issue**: Pipeline run fails at model upload stage

**Solution**:

. Verify MinIO is accessible from pipeline pods
. Check MinIO credentials in pipeline parameters (`aws_access_key_id`, `aws_secret_access_key`)
. Confirm `inference` bucket exists in MinIO
. Verify `aws_s3_endpoint` parameter is correct
. Common fix: Test MinIO connectivity from pipeline pod

**Issue**: Schedule creates runs but they don't execute

**Solution**:

. Check pipeline server pods are running: `oc get pods -n ai-edge-project | grep ds-pipeline`
. Verify scheduledworkflow pod is operational: `oc logs ds-pipeline-scheduledworkflow-<xxx> -n ai-edge-project`
. Check resource quotas: `oc get resourcequota -n ai-edge-project`
. Common fix: Restart pipeline server pods

**Issue**: Pipeline validation fails (models not deployed)

**Solution**:

. This is expected behavior when new models don't outperform existing ones
. Check validation logs to see performance comparison
. If models should have improved, verify training data quality
. Ensure test dataset is representative
. Common fix: Run training notebooks manually to verify model quality
