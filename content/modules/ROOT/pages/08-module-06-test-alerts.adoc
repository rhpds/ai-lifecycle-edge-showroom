== Exercise 6.1: Test battery health alerts

The Battery Monitoring System dashboard integrates AI predictions with real-time telemetry to provide actionable alerts. Test the alert system by simulating battery stress conditions.

=== Understanding the alerting architecture

**Data Flow:**

1. **Battery Simulator** → Generates realistic telemetry (voltage, temperature, charge level)
2. **MQTT Broker** → Central messaging hub for sensor data
3. **Data Processors** → Route data to storage (InfluxDB) and visualization (WebSocket)
4. **BMS Dashboard** → Displays real-time telemetry and triggers AI inference
5. **Model Servers** → Receive telemetry, return predictions
6. **Alert System** → Notifications when stress detected or failure imminent

**Alert Triggers:**

* **Stress Detection** - Score > 0.5 indicates STRESSED battery (immediate risk)
* **Time to Failure** - Prediction < 48 hours triggers preventive maintenance alert
* **Telemetry Thresholds** - Voltage, temperature, or charge level outside safe ranges

----
https://bms-dashboard-microshift-001.{openshift_cluster_ingress_domain}
----

The dashboard displays:

* **Real-time Telemetry** - Voltage, temperature, state of charge graphs
* **AI Prediction Panel** - Stress detection status and time-to-failure estimate
* **Alert Indicators** - Visual warnings when risks detected
* **Auto-Inference Toggle** - Enable/disable automatic prediction updates

=== Enable automatic inference

Configure the dashboard to continuously query the inference endpoints:

. Locate the **Auto-Inference** toggle in the AI Prediction Panel
. Ensure it is **enabled** (toggle should be ON)
. Verify **Interval**: 30 seconds (configurable)

When enabled, the dashboard automatically:

* Sends current telemetry to both inference endpoints every 30 seconds
* Receives stress detection score and time-to-failure prediction
* Updates the prediction panel with latest results
* Triggers alerts if thresholds are exceeded

=== Simulate battery stress

The battery simulator allows you to manually trigger stress conditions for testing:

**Option 1: Via Dashboard Controls (if available)**

. Locate battery simulation controls in the dashboard
. Adjust parameters to create stress conditions:
+
* Increase **Temperature** above 45°C
* Decrease **Voltage** below 3.0V
* Set **Discharge Rate** to high values
* Increase **Cycle Count** to simulate aging battery

**Option 2: Via Battery Simulation API**

Access the battery simulation control endpoint:

[source,bash,subs="attributes+"]
----
https://battery-simulation-microshift-001.{openshift_cluster_ingress_domain}
----

Use the API to adjust simulation parameters:

[source,bash]
----
curl -X POST https://battery-simulation-microshift-001.{openshift_cluster_ingress_domain}/api/scenario \
  -H "Content-Type: application/json" \
  -d '{
    "temperature": 50,
    "voltage": 2.8,
    "stateOfCharge": 15,
    "dischargeRate": 2.5
  }'
----

These values simulate a battery under extreme stress.

=== Observe stress detection alert

After simulating stress conditions, watch the dashboard:

. **Telemetry Graphs** - Should show stressed values (high temperature, low voltage)
. **AI Prediction Panel** - Updates after next inference cycle (within 30 seconds)
. **Stress Detection Status** - Changes from "NORMAL" to "STRESSED"
. **Stress Score** - Displays value > 0.5
. **Alert Indicator** - Visual warning appears (red icon or notification)

Expected alert message:
----
⚠️ BATTERY STRESS DETECTED
Stress Score: 0.87
Action: Investigate battery condition immediately
----

=== Observe time-to-failure prediction

Continue monitoring the dashboard:

. **Time to Failure Panel** - Updates with prediction
. **Estimated Hours** - Shows remaining operational life
. **Alert Threshold** - If < 48 hours, maintenance alert appears

Expected prediction display:
----
Predicted Time Before Failure: 12.3 hours
Battery should be replaced within: 0.5 days
⚠️ MAINTENANCE REQUIRED SOON
----

=== Reset simulation to normal

Return the battery to normal operating conditions:

[source,bash]
----
curl -X POST https://battery-simulation-microshift-001.{openshift_cluster_ingress_domain}/api/scenario \
  -H "Content-Type: application/json" \
  -d '{
    "temperature": 25,
    "voltage": 3.7,
    "stateOfCharge": 80,
    "dischargeRate": 0.5
  }'
----

Observe the dashboard:

. **Telemetry Graphs** - Return to normal values
. **Stress Detection Status** - Changes from "STRESSED" to "NORMAL"
. **Stress Score** - Drops below 0.5
. **Alert Indicators** - Warnings clear
. **Time to Failure** - Prediction increases (more hours remaining)

== Exercise 6.2: Monitor real-time inference execution

Observe how the robot's model servers process inference requests in real-time. This exercise demonstrates the behind-the-scenes AI inference operations that power the dashboard predictions.

=== Understanding inference monitoring

**Why Monitor Inference Logs:**

* **Performance analysis** - Track inference latency and throughput
* **Debugging** - Identify issues with model loading or predictions
* **Validation** - Confirm models are receiving correct input data
* **Audit trail** - Maintain records of AI decision-making for compliance

Model server logs show:

* Incoming inference requests with input tensors
* Model execution and prediction generation
* Response payload with prediction results
* Request timing and performance metrics

=== Access robot model server logs

Connect to the robot and view stress detection model server logs:

. SSH to bastion host:
+
[source,bash,subs="attributes+"]
----
{bastion_ssh_command}
----
+
**Password:** `{bastion_ssh_password}`

. Connect to robot VM:
+
[source,bash,subs="attributes+"]
----
virtctl ssh cloud-user@vmi/microshift -n microshift-001 --identity-file=/home/lab-user/.ssh/microshift-001
----

. View stress detection inference logs:
+
[source,bash]
----

The `-f` flag streams logs in real-time (similar to `tail -f`).

=== Trigger inference requests

Keep the log stream open. In a separate browser window, access the BMS dashboard and enable auto-inference (if not already enabled):

[source,bash,subs="attributes+"]
----

----

Ensure **Auto-Inference** toggle is ON with 30-second interval.

=== Observe inference request processing

Watch the log stream for inference activity. You should see entries every 30 seconds:

**Expected log patterns:**

----
[INFO] Received inference request
[INFO] Model: stress-detection, Version: 1
[INFO] Input tensor shape: [1, 9], datatype: FP32
[INFO] Input data: [0.52, 0.48, 0.65, 0.42, 0.55, 0.38, 0.50, 0.61, 0.44]
[INFO] Inference execution started
[INFO] Prediction completed in 12ms
[INFO] Output: [0.32]
[INFO] Response sent successfully
----

**Key information to observe:**

* **Request frequency** - Should match auto-inference interval (every 30 seconds)
* **Input shape** - `[1, 9]` (1 sample, 9 features for stress detection)
* **Input values** - Normalized feature values (0-1 range)
* **Inference latency** - Typical: 10-50ms for edge inference
* **Prediction output** - Single value (stress score between 0 and 1)

== Troubleshooting

**Issue**: Dashboard shows "Connection Failed" or no real-time data

**Solution**:

. Verify all battery-demo pods are running: `oc get pods -n battery-demo`
. Check WebSocket endpoint accessibility from browser
. Verify MQTT broker is operational: `oc logs -n battery-demo -l app=mosquitto-mqtt-broker`
. Check battery-simulation pod is generating data: `oc logs -n battery-demo -l app=battery-simulation`
. Common fix: Restart battery-demo pods: `oc rollout restart deployment -n battery-demo`

**Issue**: Auto-inference not updating predictions

**Solution**:

. Verify auto-inference toggle is enabled in dashboard
. Check browser console for JavaScript errors (F12 → Console)
. Verify inference endpoints are accessible from browser
. Test endpoints manually using curl from your laptop
. Common fix: Clear browser cache and reload dashboard

**Issue**: Inference endpoints return 404 or timeout errors

**Solution**:

. Verify model server pods are running: `oc get pods -n inference`
. Check routes exist: `oc get routes -n inference`
. Verify route URLs match dashboard configuration
. Test internal endpoint from workbench: `curl http://stress-detection-predictor.ai-edge-project.svc.cluster.local:8888/v2/models/stress-detection`
. Common fix: Recreate routes or check network policies

**Issue**: Predictions always return same value regardless of input

**Solution**:

. Verify models are loaded correctly in server pods
. Check model server logs for errors: `oc logs -n inference <pod-name>`
. Ensure input features are properly normalized (0-1 range)
. Verify telemetry data has reasonable variation
. Common fix: Retrain models with fresh data

**Issue**: Battery simulation API returns errors

**Solution**:

. Verify battery-simulation pod is running: `oc get pods -n battery-demo | grep battery-simulation`
. Check pod logs for errors: `oc logs -n battery-demo -l app=battery-simulation`
. Verify route exists: `oc get route battery-simulation -n battery-demo`
. Test API endpoint accessibility: `curl https://battery-simulation-microshift-001.{openshift_cluster_ingress_domain}/api/health`
. Common fix: Restart battery-simulation deployment
