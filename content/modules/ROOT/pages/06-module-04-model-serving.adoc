== Exercise 4.1: Review Stress Detection model server

Model servers provide real-time inference capabilities by loading trained models and exposing REST API endpoints. The Stress Detection server validates newly trained models before deployment to robots.

=== Understanding model serving in RHOAI

**Why Model Serving on SNO?**

Although models run on robots for production inference, the SNO environment also serves models for:

* **Validation** - Test newly trained models before deploying to robots
* **Comparison** - Benchmark new models against existing ones
* **Development** - Quick testing during model development

**KServe Deployment Modes:**

* **Serverless** - Auto-scaling with Knative (not used here)
* **RawDeployment** - Manual configuration, predictable resources (used for edge)

RawDeployment provides full control without auto-scaling overhead, ideal for resource-constrained SNO.

== Exercise 4.3: Query model endpoints

Test the model inference endpoints to verify they respond correctly to prediction requests. You'll use a Jupyter notebook to send test data and receive predictions.

=== Understanding inference requests

Model inference uses HTTP POST requests with JSON payloads following the KServe v2 protocol.

**Endpoint Format:**

[source,text]
----
POST http://<service-name>.ai-edge-project.svc.cluster.local:8888/v2/models/<model-name>/infer
----

**Request Body Structure:**

[source,json]
----
{
  "inputs": [
    {
      "name": "keras_tensor",       // Input tensor name (from model)
      "shape": [1, N],               // Batch size 1, N features
      "datatype": "FP32",            // 32-bit floating point
      "data": [...]                  // Feature values (normalized)
    }
  ]
}
----

**Input Features:**

**Stress Detection (9 features):**

* Voltage, Temperature, State of Charge, Discharge Rate
* Time Elapsed, Cycle Count, Ambient Temperature
* Usage Pattern, Health Status

**Time to Failure (5 features):**

* Temperature Delta, Voltage Delta, Resistance
* Current Delta, Capacity Fade

**Response Structure:**

[source,json]
----
{
  "outputs": [
    {
      "name": "output",
      "shape": [1, 1],
      "datatype": "FP32",
      "data": [0.87]              // Prediction (stress score or hours)
    }
  ]
}
----

For Stress Detection: values > 0.5 indicate STRESSED, ≤ 0.5 indicate NORMAL

For Time to Failure: value represents predicted hours until battery failure

[#troubleshooting]
== Troubleshooting

**Issue**: Model server pod fails to start with "ImagePullBackOff"

**Solution**:

. Check pod status: `oc describe pod <pod-name> -n ai-edge-project`
. Verify image name in ServingRuntime configuration
. Ensure network connectivity for image registry
. Common fix: Wait for automatic retry or check image pull secrets

**Issue**: Model server logs show "Failed to load model" error

**Solution**:

. Verify MinIO is accessible from model server pod
. Check DataConnection credentials are correct
. Confirm model files exist in MinIO bucket
. Verify model path matches InferenceService configuration
. Common fix: Re-upload model files to MinIO with correct structure

**Issue**: Inference endpoint returns 404 error

**Solution**:

. Verify InferenceService is in Ready state: `oc get inferenceservice -n ai-edge-project`
. Check service name matches endpoint URL
. Ensure model name in URL matches InferenceService name
. Common fix: Wait for InferenceService to fully deploy (check READY column)

**Issue**: Query notebook fails with "Connection timeout" error

**Solution**:

. Verify model server pods are running: `oc get pods -n ai-edge-project`
. Check workbench can resolve cluster DNS: Run `nslookup stress-detection-predictor.ai-edge-project.svc.cluster.local` from workbench terminal
. Verify network policies allow traffic between namespaces
. Common fix: Restart workbench and retry

**Issue**: Model predictions return NaN or unreasonable values

**Solution**:

. Verify input data is properly normalized (typically 0-1 range)
. Check feature count matches model input shape (9 for Stress, 5 for TTF)
. Ensure data types are correct (FP32)
. Verify model was trained and exported correctly
. Common fix: Retrain model with proper data preprocessing
